📖 Introduction

L'optimisation des requêtes SQL Server est un enjeu critique pour garantir des performances optimales de vos applications. Ce guide vous propose une approche structurée pour identifier, diagnostiquer et résoudre les problèmes de performance les plus courants.

Que vous soyez confronté à des requêtes lentes, des blocages, des deadlocks, ou des problèmes de mémoire, vous trouverez ici des solutions concrètes et des bonnes pratiques pour améliorer l'efficacité de votre base de données.

⚠️ Problématiques Courantes de Performance

Voici les 10 problèmes les plus fréquents affectant les performances des requêtes SQL Server, classés par fréquence et impact.

1. Plan d'exécution sous-optimal Impact Élevé

SQL Server génère un plan d'exécution coûteux en CPU, I/O ou temps, alors qu'un plan plus efficace existe.

Symptômes :

  • Temps d'exécution anormalement long
  • I/O élevé (lectures logiques/physiques)
  • CPU élevé (opérations de tri, jointures coûteuses)
  • Présence d'opérateurs coûteux : Table Scan, Sort, Hash Match

Causes possibles :

  • Statistiques obsolètes
  • Paramètre Sniffing
  • Manque ou mauvaise conception d'index
  • Requêtes complexes (sous-requêtes, CTEs)
  • Heuristiques du Query Optimizer

2. Blocking (Blocage) Impact Élevé

Une requête attend qu'une autre libère un verrou sur une ressource (ligne, page, table).

Symptômes :

  • Requêtes en attente (WAIT dans sys.dm_exec_requests)
  • Timeout si LOCK_TIMEOUT est configuré
  • Lenteur globale du système

Causes possibles :

  • Transactions longues
  • Niveau d'isolation élevé (SERIALIZABLE, REPEATABLE READ)
  • Verrous implicites (WITH (UPDLOCK, XLOCK))
  • Conflits de verrous (verrou partagé vs exclusif)

3. Deadlocks Impact Élevé

Deux requêtes s'attendent mutuellement pour acquérir des verrous, provoquant un blocage infini. SQL Server tue une des requêtes (victime) pour résoudre le deadlock.

Symptômes :

  • Erreur 1205 : Deadlock encountered...
  • Requêtes annulées sans raison apparente
  • Logs de deadlocks dans system_health

Causes possibles :

  • Ordre d'accès aux ressources différent entre requêtes
  • Transactions longues
  • Verrous incompatibles

4. Spill to Disk (Déversement sur disque) Impact Moyen

SQL Server manque de mémoire pour exécuter une opération (tri, jointure, agrégation) et doit utiliser le tempdb comme mémoire temporaire, ce qui ralentit considérablement la requête.

Symptômes :

  • Messages dans le plan d'exécution : Warning: Operator used tempdb to spill data
  • I/O élevé sur tempdb
  • Lenteur soudaine pour les requêtes avec tri/jointure

Causes possibles :

  • Mémoire insuffisante (max server memory trop bas)
  • Requêtes gourmandes en mémoire (ORDER BY, GROUP BY, HASH JOIN)
  • tempdb mal configuré (un seul fichier de données)
  • Parallelisme excessif

5. I/O Élevé (Lectures disque) Impact Moyen

La requête effectue trop de lectures physiques depuis le disque, ce qui la ralentit.

Symptômes :

  • Temps d'exécution long (attente sur PAGEIOLATCH_*)
  • I/O élevé dans sys.dm_exec_query_stats (logical_reads, physical_reads)
  • Attentes sur le disque (PAGEIOLATCH_SH, PAGEIOLATCH_EX)

Causes possibles :

  • Manque d'index (Table Scan ou Clustered Index Scan)
  • Index non sélectifs
  • Buffer Pool saturé
  • Disques lents (HDD au lieu de SSD)
  • Fragmentation des index

6. CPU Élevé Impact Moyen

La requête consomme trop de CPU, ce qui ralentit le serveur et peut impacter d'autres requêtes.

Symptômes :

  • CPU élevé dans sys.dm_exec_query_stats (cpu_time)
  • Attentes sur SOS_SCHEDULER_YIELD (CPU saturé)
  • Lenteur globale du serveur

Causes possibles :

  • Opérateurs coûteux en CPU (SORT, HASH MATCH, COMPUTE SCALAR)
  • Fonctions scalaires (UPPER, LOWER, SUBSTRING)
  • Boucles (WHILE, cursors)
  • Parallelisme excessif

7. Attentes (Waits) Impact Moyen

Les requêtes attendent une ressource (verrou, CPU, I/O, mémoire, etc.), ce qui ralentit leur exécution.

Symptômes :

  • Requêtes lentes sans raison apparente
  • Attentes visibles dans sys.dm_os_wait_stats ou sys.dm_exec_requests

Causes possibles :

  • PAGEIOLATCH_* : Lecture depuis le disque (I/O lent)
  • LCK_* : Attente de verrou (blocking)
  • SOS_SCHEDULER_YIELD : CPU saturé
  • CXPACKET : Attente de synchronisation entre threads

8. Problèmes de tempdb Impact Moyen

tempdb est une base système utilisée pour les tables temporaires, les opérations de tri, les jointures de hachage, etc. Les problèmes sur tempdb peuvent ralentir toutes les requêtes.

Symptômes :

  • Lenteur des requêtes utilisant tempdb
  • Attentes sur PFS ou GAM
  • I/O élevé sur tempdb
  • Messages de spill to disk dans les plans d'exécution

Causes possibles :

  • Un seul fichier de données (tempdb.mdf)
  • Fichiers de données trop petits (autogrowth fréquent)
  • Disques lents pour tempdb
  • Utilisation excessive (requêtes avec ORDER BY, GROUP BY)

9. Problèmes de Concurrence Impact Moyen

Trop de requêtes s'exécutent simultanément, ce qui entraîne une contention sur les ressources (CPU, I/O, mémoire), des blocages et des deadlocks.

Symptômes :

  • Temps de réponse élevé pour toutes les requêtes
  • Attentes fréquentes (LCK_*, SOS_SCHEDULER_YIELD)
  • Deadlocks fréquents
  • CPU à 100% en permanence

Causes possibles :

  • Trop de connexions (pool de connexions mal configuré)
  • Requêtes longues
  • Parallelisme excessif
  • Verrous incompatibles
  • Ressources limitées (CPU, mémoire, I/O)

10. Problèmes de Compilation/Recompilation Impact Faible

Les requêtes sont recompilées trop souvent, ce qui consomme du CPU et ralentit l'exécution.

Symptômes :

  • Temps de compilation élevé (visible dans sys.dm_exec_query_stats)
  • CPU élevé sans I/O important
  • Plans différents pour la même requête

Causes possibles :

  • Statistiques obsolètes
  • Changements de schéma
  • Variables locales dans WHERE
  • Option RECOMPILE forcée
  • Paramètre Sniffing
  • Requêtes ad-hoc

🔧 Outils de Diagnostic

Voici les outils essentiels pour diagnostiquer les problèmes de performance dans SQL Server.

📊 Plan d'Exécution

Affiche le plan d'exécution estimé ou réel d'une requête, avec les coûts relatifs, les opérateurs utilisés et les warnings (ex. : spill to disk).

Commandes :

SET SHOWPLAN_TEXT ON
SET STATISTICS PROFILE ON
SET STATISTICS XML ON

📈 DMV (Dynamic Management Views)

Les DMV fournissent des informations en temps réel sur les performances du serveur, des requêtes, des index, etc.

DMV utiles :

sys.dm_exec_query_stats
sys.dm_exec_requests
sys.dm_os_wait_stats
sys.dm_tran_locks
sys.dm_io_virtual_file_stats

🔍 SQL Server Profiler

Outil graphique pour capturer et analyser les événements (requêtes, verrous, deadlocks, etc.) en temps réel.

Événements clés :

SQL:BatchStarting
SQL:BatchCompleted
Lock:Acquired
Lock:Released
Deadlock graph

📉 Extended Events

Alternative légère à Profiler, permettant de capturer des événements spécifiques avec un impact minimal sur les performances.

Sessions utiles :

system_health
Query Performance

📊 Query Store

Fonctionnalité intégrée (SQL Server 2016+) qui capture les requêtes, leurs plans d'exécution et leurs performances au fil du temps.

Commandes :

ALTER DATABASE [Nom] SET QUERY_STORE = ON

🛠️ Database Engine Tuning Advisor

Outil qui analyse les requêtes et recommande des index, statistiques ou partitions pour améliorer les performances.

✅ Solutions et Bonnes Pratiques

Voici des solutions concrètes pour chaque problématique, ainsi que des bonnes pratiques générales pour optimiser vos requêtes SQL Server.

1. Optimiser le Plan d'Exécution

🔹 Mettre à jour les statistiques

Les statistiques aident le Query Optimizer à générer des plans efficaces.

-- Mettre à jour les statistiques d'une table
UPDATE STATISTICS [Schema].[Table] WITH FULLSCAN;

-- Mettre à jour toutes les statistiques de la base
EXEC sp_updatestats;

Bonnes pratiques :

  • Activer AUTO_UPDATE_STATISTICS (activé par défaut).
  • Pour les grandes tables, utiliser WITH SAMPLE 50 PERCENT pour éviter un FULLSCAN coûteux.
  • Mettre à jour manuellement après des chargements massifs.

🔹 Éviter le Paramètre Sniffing

Le Paramètre Sniffing peut conduire à des plans sous-optimaux si la première valeur du paramètre est atypique.

-- Utiliser OPTION (RECOMPILE)
EXEC sp_executesql N'SELECT * FROM Clients WHERE Ville = @Ville', N'@Ville VARCHAR(50)', @Ville = 'Paris' WITH RECOMPILE;

-- Utiliser OPTION (OPTIMIZE FOR UNKNOWN)
SELECT * FROM Clients WHERE Ville = @Ville OPTION (OPTIMIZE FOR UNKNOWN);

🔹 Ajouter ou optimiser les index

Les index améliorent les performances des requêtes en permettant des Index Seek au lieu de Table Scan.

-- Index simple sur une colonne
CREATE INDEX IX_Clients_Ville ON Clients(Ville);

-- Index composite
CREATE INDEX IX_Clients_Ville_Nom ON Clients(Ville, Nom);

-- Index couvrant (INCLUDE)
CREATE INDEX IX_Clients_Ville ON Clients(Ville) INCLUDE (Nom, Prénom);

-- Index filtré
CREATE INDEX IX_Clients_Ville_Paris ON Clients(Ville) WHERE Ville = 'Paris';
Astuce : Utilisez le Database Engine Tuning Advisor pour identifier les index manquants.

🔹 Réécrire la requête

Évitez les fonctions sur les colonnes dans les clauses WHERE, car elles empêchent l'utilisation des index.

-- ❌ Mauvaise pratique (ne peut pas utiliser l'index sur Ville)
SELECT * FROM Clients WHERE UPPER(Ville) = 'PARIS';

-- ✅ Bonne pratique
SELECT * FROM Clients WHERE Ville = 'Paris' COLLATE SQL_Latin1_General_CP1_CI_AS;
-- ❌ Sous-requête lente
SELECT * FROM Clients WHERE Id IN (SELECT ClientId FROM Commandes WHERE Montant > 1000);

-- ✅ JOIN plus rapide
SELECT DISTINCT c.* FROM Clients c
JOIN Commandes co ON c.Id = co.ClientId
WHERE co.Montant > 1000;

2. Résoudre le Blocking

🔹 Réduire la durée des transactions

Les transactions longues augmentent le risque de blocking.

-- ❌ Transaction trop longue
BEGIN TRANSACTION;
UPDATE Compte SET Solde = Solde - 100 WHERE Id = 1;
-- Ne pas garder la transaction ouverte
COMMIT;
Astuce : Commitez les transactions dès que possible.

🔹 Choisir le bon niveau d'isolation

Le niveau d'isolation détermine les verrous acquis par une transaction.

Niveau Verrous Problèmes Utilisation
READ UNCOMMITTED Aucun (dirty reads) Données non validées Rapports, analyses
READ COMMITTED (défaut) Verrous partagés Blocking possible Requêtes standard
REPEATABLE READ Verrous partagés jusqu'à la fin de la transaction Blocking accru Peu utilisé
SERIALIZABLE Verrous partagés + verrous de plage Blocking maximal Transactions critiques
SNAPSHOT Aucune (versioning) Overhead mémoire Requêtes en lecture seule
-- Activer SNAPSHOT ISOLATION au niveau de la base
ALTER DATABASE MaBase SET ALLOW_SNAPSHOT_ISOLATION ON;

-- Utiliser dans une requête
SET TRANSACTION ISOLATION LEVEL SNAPSHOT;
BEGIN TRANSACTION;
SELECT * FROM Clients WHERE Ville = 'Paris';
COMMIT;

🔹 Utiliser des hints de verrouillage

Les hints de verrouillage permettent de contrôler les verrous acquis par une requête.

-- Ignorer les verrous (risque de dirty reads)
SELECT * FROM Clients WITH (NOLOCK) WHERE Ville = 'Paris';

-- Prendre un verrou de mise à jour
SELECT * FROM Clients WITH (UPDLOCK) WHERE Ville = 'Paris';
Attention : NOLOCK peut entraîner des dirty reads (lecture de données non validées).

3. Résoudre les Deadlocks

🔹 Analyser les graphes de deadlock

Les graphes de deadlock (capturés dans system_health) montrent quelles requêtes et quels verrous sont en conflit.

-- Exemple de graphe de deadlock (XML)
<deadlock>
  <victim-list>
    <victimProcess id="process1" />
  </victim-list>
  <process-list>
    <process id="process1" waitresource="RID: 6:1:12345" waittype="LR" />
    <process id="process2" waitresource="RID: 6:1:67890" waittype="LR" />
  </process-list>
</deadlock>
Astuce : Utilisez SQL Server Profiler ou Extended Events pour capturer les deadlocks.

🔹 Réécrire les requêtes pour éviter les deadlocks

Assurez-vous que toutes les requêtes accèdent aux tables dans le même ordre.

-- ❌ Risque de deadlock
-- Requête 1 : UPDATE TableA THEN TableB
-- Requête 2 : UPDATE TableB THEN TableA

-- ✅ Toujours accéder dans l'ordre TableA -> TableB
BEGIN TRANSACTION;
UPDATE TableA SET ... WHERE Id = 1;
UPDATE TableB SET ... WHERE Id = 1;
COMMIT;

🔹 Utiliser TRY/CATCH pour gérer les deadlocks

Capturez les deadlocks et réessayez automatiquement.

BEGIN TRY
BEGIN TRANSACTION;
UPDATE Compte SET Solde = Solde - 100 WHERE Id = 1;
UPDATE Compte SET Solde = Solde + 100 WHERE Id = 2;
COMMIT;
END TRY
BEGIN CATCH
IF ERROR_NUMBER() = 1205 -- Deadlock
BEGIN
ROLLBACK;
-- Réessayer après un délai
WAITFOR DELAY '00:00:01';
-- Réexécuter la transaction
END
ELSE
BEGIN
ROLLBACK;
-- Gérer l'erreur
END
END CATCH;

🔹 Utiliser SNAPSHOT ISOLATION

SNAPSHOT ISOLATION évite les verrous en lecture, réduisant ainsi les risques de deadlock.

ALTER DATABASE MaBase SET ALLOW_SNAPSHOT_ISOLATION ON;

SET TRANSACTION ISOLATION LEVEL SNAPSHOT;
BEGIN TRANSACTION;
SELECT * FROM Clients WHERE Ville = 'Paris';
-- Pas de verrou partagé, donc pas de blocking
COMMIT;

4. Éviter le Spill to Disk

🔹 Augmenter la mémoire allouée à SQL Server

Si SQL Server manque de mémoire, il utilise tempdb comme mémoire temporaire, ce qui ralentit les requêtes.

-- Vérifier la mémoire actuelle
SELECT name, value, value_in_use
FROM sys.configurations
WHERE name = 'max server memory';

-- Modifier (ex. : 16 Go)
EXEC sp_configure 'show advanced options', 1;
RECONFIGURE;
EXEC sp_configure 'max server memory', 16384; -- 16 Go en Mo
RECONFIGURE;
Astuce : Laissez 10-20% de RAM libre pour le système d'exploitation.

🔹 Optimiser tempdb

tempdb doit être configuré correctement pour éviter les contentions.

-- Ajouter des fichiers de données à tempdb (1 par CPU logique, jusqu'à 8)
ALTER DATABASE tempdb
ADD FILE (NAME = tempdb2, FILENAME = 'C:\SQLData\tempdb2.ndf', SIZE = 1GB, FILEGROWTH = 512MB);

-- Pré-allouer de l'espace
ALTER DATABASE tempdb
MODIFY FILE (NAME = tempdb, SIZE = 10GB, FILEGROWTH = 1GB);
Astuce : Placez tempdb sur des disques rapides (SSD, RAID 10).

🔹 Réduire la mémoire utilisée par les requêtes

Évitez les opérations gourmandes en mémoire comme ORDER BY, GROUP BY, ou HASH JOIN sur de grands jeux de données.

-- ❌ Requête avec spill
SELECT * FROM (
SELECT * FROM Clients
WHERE Ville = 'Paris'
) AS c
JOIN Commandes co ON c.Id = co.ClientId
ORDER BY co.Montant DESC;

-- ✅ Découper en deux requêtes
-- 1. Récupérer les clients de Paris
SELECT Id, Nom INTO #ClientsParis FROM Clients WHERE Ville = 'Paris';

-- 2. Joindre avec Commandes (avec un jeu de données réduit)
SELECT * FROM #ClientsParis c
JOIN Commandes co ON c.Id = co.ClientId
ORDER BY co.Montant DESC;

DROP TABLE #ClientsParis;

🔹 Limiter le parallelisme

Le parallelisme peut consommer trop de mémoire et provoquer des spills.

-- Configurer Cost Threshold for Parallelism (par défaut : 5)
EXEC sp_configure 'show advanced options', 1;
RECONFIGURE;
EXEC sp_configure 'cost threshold for parallelism', 50;
RECONFIGURE;

-- Limiter le MAXDOP (degré de parallelisme)
EXEC sp_configure 'max degree of parallelism', 4; -- 4 CPU max
RECONFIGURE;

5. Réduire l'I/O Élevé

🔹 Ajouter des index

Les index permettent d'éviter les Table Scan coûteux.

-- Index sur une colonne utilisée dans WHERE
CREATE INDEX IX_Clients_Ville ON Clients(Ville);

-- Index composite pour WHERE + ORDER BY
CREATE INDEX IX_Clients_Ville_Nom ON Clients(Ville, Nom);

-- Index couvrant pour éviter les Key Lookup
CREATE INDEX IX_Clients_Ville ON Clients(Ville) INCLUDE (Nom, Prénom);

🔹 Optimiser les requêtes

Évitez les SELECT * et utilisez des jointures efficaces.

-- ❌ SELECT *
SELECT * FROM Clients WHERE Ville = 'Paris';

-- ✅ Sélectionner uniquement les colonnes nécessaires
SELECT Id, Nom, Prénom FROM Clients WHERE Ville = 'Paris';

🔹 Réduire la fragmentation des index

La fragmentation des index peut dégrader les performances.

-- Vérifier la fragmentation
SELECT
OBJECT_NAME(ind.OBJECT_ID) AS TableName,
ind.name AS IndexName,
indexstats.avg_fragmentation_in_percent
FROM sys.dm_db_index_physical_stats(DB_ID(), NULL, NULL, NULL, NULL) indexstats
INNER JOIN sys.indexes ind ON ind.object_id = indexstats.object_id AND ind.index_id = indexstats.index_id
WHERE indexstats.avg_fragmentation_in_percent > 30;

-- Réorganiser (fragmentation entre 10% et 30%)
ALTER INDEX IX_Clients_Ville ON Clients REORGANIZE;

-- Reconstruire (fragmentation > 30%)
ALTER INDEX IX_Clients_Ville ON Clients REBUILD WITH (ONLINE = OFF);

🔹 Utiliser des disques rapides

Les disques lents (HDD) peuvent causer des goulots d'étranglement en I/O.

Astuce : Utilisez des SSD ou des disques en RAID 10 pour les bases de données.

6. Réduire l'Utilisation du CPU

🔹 Éviter les fonctions scalaires dans WHERE

Les fonctions sur les colonnes dans WHERE empêchent l'utilisation des index.

-- ❌ Mauvaise pratique
SELECT * FROM Clients WHERE UPPER(Ville) = 'PARIS';

-- ✅ Bonne pratique
SELECT * FROM Clients WHERE Ville = 'Paris' COLLATE SQL_Latin1_General_CP1_CI_AS;

🔹 Remplacer les cursors par des requêtes ensemblistes

Les cursors sont lents et consomment beaucoup de CPU.

-- ❌ Cursor lent
DECLARE @Id INT;
DECLARE cur CURSOR FOR SELECT Id FROM Clients WHERE Ville = 'Paris';
OPEN cur;
FETCH NEXT FROM cur INTO @Id;
WHILE @@FETCH_STATUS = 0
BEGIN
UPDATE Commandes SET Statut = 'Traité' WHERE ClientId = @Id;
FETCH NEXT FROM cur INTO @Id;
END
CLOSE cur;
DEALLOCATE cur;

-- ✅ Requête ensembliste
UPDATE co
SET Statut = 'Traité'
FROM Commandes co
JOIN Clients c ON co.ClientId = c.Id
WHERE c.Ville = 'Paris';

🔹 Limiter le parallelisme

Le parallelisme peut consommer trop de CPU.

-- Configurer Cost Threshold for Parallelism
EXEC sp_configure 'show advanced options', 1;
RECONFIGURE;
EXEC sp_configure 'cost threshold for parallelism', 50;
RECONFIGURE;

-- Limiter le MAXDOP au niveau d'une requête
SELECT * FROM Clients WHERE Ville = 'Paris' OPTION (MAXDOP 2);

7. Résoudre les Attentes (Waits)

🔹 Identifier les attentes dominantes

Utilisez les DMV pour identifier les attentes les plus fréquentes.

-- Top 10 des attentes
SELECT
wait_type,
waiting_tasks_count,
wait_time_ms,
max_wait_time_ms,
signal_wait_time_ms
FROM sys.dm_os_wait_stats
ORDER BY wait_time_ms DESC;

🔹 Résoudre les attentes spécifiques

Attente Cause Solution
PAGEIOLATCH_SH I/O lent Optimiser les index, ajouter de la mémoire, utiliser des SSD
LCK_M_S Blocking Réduire la durée des transactions, utiliser SNAPSHOT ISOLATION
SOS_SCHEDULER_YIELD CPU saturé Limiter le parallelisme, optimiser les requêtes
CXPACKET Parallelisme excessif Réduire le MAXDOP
ASYNC_NETWORK_IO Client lent Optimiser l'application (ex. : SET NOCOUNT ON)
WRITELOG Log lent Placer le log sur un disque rapide, commiter moins souvent

8. Optimiser tempdb

🔹 Configurer tempdb correctement

tempdb doit avoir plusieurs fichiers de données pour éviter les contentions.

-- Ajouter des fichiers de données à tempdb (1 par CPU logique, jusqu'à 8)
USE master;
GO
ALTER DATABASE tempdb
ADD FILE (NAME = tempdb2, FILENAME = 'C:\SQLData\tempdb2.ndf', SIZE = 1GB, FILEGROWTH = 512MB);
ALTER DATABASE tempdb
ADD FILE (NAME = tempdb3, FILENAME = 'C:\SQLData\tempdb3.ndf', SIZE = 1GB, FILEGROWTH = 512MB);
ALTER DATABASE tempdb
ADD FILE (NAME = tempdb4, FILENAME = 'C:\SQLData\tempdb4.ndf', SIZE = 1GB, FILEGROWTH = 512MB);
Astuce : Désactivez l'autogrowth pour éviter la fragmentation.

🔹 Éviter les variables de table

Les variables de table (@table) n'ont pas d'index et peuvent causer des performances médiocres.

-- ❌ Variable de table lente
DECLARE @Clients TABLE (Id INT, Nom VARCHAR(50)); INSERT INTO @Clients SELECT Id, Nom FROM Clients WHERE Ville = 'Paris'; SELECT * FROM @Clients WHERE Nom LIKE 'D%';

-- ✅ Table temporaire
CREATE TABLE #Clients (Id INT, Nom VARCHAR(50)); CREATE INDEX IX_Nom ON #Clients(Nom); INSERT INTO #Clients SELECT Id, Nom FROM Clients WHERE Ville = 'Paris'; SELECT * FROM #Clients WHERE Nom LIKE 'D%'; DROP TABLE #Clients;

9. Gérer la Concurrence

🔹 Limiter le nombre de connexions

Trop de connexions peuvent saturer le serveur.

-- Configurer le pool de connexions dans l'application
-- Exemple en C# (ADO.NET)
using (SqlConnection connection = new SqlConnection(connectionString)) { // Max Pool Size = 100
connectionString += ";Max Pool Size=100"; }

-- Limiter les connexions au niveau de SQL Server
EXEC sp_configure 'show advanced options', 1;
RECONFIGURE;
EXEC sp_configure 'user connections', 100; -- Limiter à 100 connexions
RECONFIGURE;

🔹 Utiliser Resource Governor

Resource Governor permet de limiter les ressources (CPU, mémoire, I/O) pour différents groupes de travail.

-- Créer un pool de ressources
CREATE RESOURCE POOL AdHocPool WITH ( MAX_CPU_PERCENT = 30, CAP_CPU_PERCENT = 50, MAX_MEMORY_PERCENT = 20 );

-- Créer un groupe de travail
CREATE WORKLOAD GROUP AdHocGroup USING AdHocPool;

🔹 Utiliser Query Store

Query Store capture les requêtes, leurs plans d'exécution et leurs performances au fil du temps.

-- Activer Query Store
ALTER DATABASE MaBase SET QUERY_STORE = ON (OPERATION_MODE = READ_WRITE, CLEANUP_POLICY = (STALE_QUERY_THRESHOLD_DAYS = 30));

-- Identifier les requêtes problématiques
SELECT qt.query_sql_text, qs.execution_type_desc, qs.count_executions, qs.avg_logical_io_reads, qs.avg_duration_ms FROM sys.query_store_query q JOIN sys.query_store_query_text qt ON q.query_text_id = qt.query_text_id JOIN sys.query_store_runtime_stats qs ON q.query_id = qs.query_id ORDER BY qs.avg_duration_ms DESC;

10. Éviter les Recompilations Inutiles

🔹 Utiliser des requêtes paramétrées

Les requêtes paramétrées permettent de réutiliser les plans d'exécution.

-- ❌ Recompile à chaque exécution (variable locale)
DECLARE @Id INT = 1; SELECT * FROM Clients WHERE Id = @Id;

-- ✅ Requête paramétrée (plan réutilisable)
EXEC sp_executesql N'SELECT * FROM Clients WHERE Id = @Id', N'@Id INT', @Id = 1;

🔹 Mettre à jour les statistiques automatiquement

Les statistiques obsolètes peuvent causer des recompilations inutiles.

-- Activer la mise à jour automatique des statistiques
ALTER DATABASE MaBase SET AUTO_UPDATE_STATISTICS ON;

-- Configurer la mise à jour asynchrone
ALTER DATABASE MaBase SET AUTO_UPDATE_STATISTICS_ASYNC ON;

🔹 Utiliser OPTION (KEEP PLAN)

KEEP PLAN (SQL Server 2016+) permet de conserver le plan d'exécution même si les statistiques changent.

SELECT * FROM Clients WHERE Ville = 'Paris' OPTION (KEEP PLAN);

🔹 Nettoyer le cache des plans

Si le cache est pollué par des plans ad-hoc, vous pouvez le nettoyer.

-- Nettoyer le cache des plans (à utiliser avec prudence)
DBCC FREESYSTEMCACHE('ALL'); DBCC FREEPROCCACHE;

-- Utiliser OPTIMIZE_FOR_AD_HOC_WORKLOADS pour réduire la mémoire utilisée par les plans ad-hoc
ALTER DATABASE MaBase SET OPTIMIZE_FOR_AD_HOC_WORKLOADS ON;
Attention : Nettoyer le cache des plans peut dégrader temporairement les performances car SQL Server devra recompiler les requêtes.

📋 Checklist de Dépannage

Suivez cette checklist pour diagnostiquer et résoudre les problèmes de performance dans SQL Server.

🔹 Étape 1 : Identifier la requête lente

Utilisez les DMV ou Query Store pour identifier les requêtes les plus lentes.

-- Top 10 des requêtes les plus lentes
SELECT TOP 10
qs.execution_count,
qs.total_logical_reads,
qs.total_logical_writes,
qs.total_worker_time,
qs.total_elapsed_time,
qs.total_elapsed_time / qs.execution_count AS avg_elapsed_time,
SUBSTRING(qt.text, (qs.statement_start_offset/2) + 1,
((CASE qs.statement_end_offset
WHEN -1 THEN DATALENGTH(qt.text)
ELSE qs.statement_end_offset
END - qs.statement_start_offset)/2) + 1) AS query_text,
qp.query_plan
FROM sys.dm_exec_query_stats qs
CROSS APPLY sys.dm_exec_sql_text(qs.sql_handle) qt
CROSS APPLY sys.dm_exec_query_plan(qs.plan_handle) qp
ORDER BY qs.total_elapsed_time / qs.execution_count DESC;

🔹 Étape 2 : Analyser le plan d'exécution

Vérifiez les opérateurs coûteux et les warnings dans le plan d'exécution.

  • Cherchez les opérateurs Table Scan, Index Scan, Sort, Hash Match.
  • Vérifiez les warnings (ex. : spill to disk).
  • Utilisez SET SHOWPLAN_TEXT ON ou SET STATISTICS PROFILE ON.

🔹 Étape 3 : Vérifier les attentes

Identifiez les attentes dominantes avec les DMV.

-- Top 10 des attentes
SELECT
wait_type,
waiting_tasks_count,
wait_time_ms,
max_wait_time_ms,
signal_wait_time_ms
FROM sys.dm_os_wait_stats
ORDER BY wait_time_ms DESC;

🔹 Étape 4 : Vérifier les verrous

Identifiez les blocages avec sys.dm_tran_locks.

-- Voir les verrous actifs
SELECT
r.session_id,
r.wait_type,
r.wait_time,
r.wait_resource,
t.text AS [Query]
FROM sys.dm_exec_requests r
CROSS APPLY sys.dm_exec_sql_text(r.sql_handle) t
WHERE r.wait_type LIKE 'LCK%';

🔹 Étape 5 : Vérifier l'utilisation des ressources

Surveillez l'utilisation du CPU, de la mémoire et de l'I/O.

-- Utilisation du CPU
SELECT
SQLProcessUtilization AS [SQL Server CPU %],
SystemIdle AS [System Idle %],
100 - SystemIdle - SQLProcessUtilization AS [Other Processes CPU %]
FROM ( SELECT
record.value('(./Record/@id)[1]', 'int') AS record_id, record.value('(./Record/SchedulerMonitorEvent/SystemHealth/SystemIdle)[1]', 'int') AS SystemIdle, record.value('(./Record/SchedulerMonitorEvent/SystemHealth/ProcessUtilization)[1]', 'int') AS SQLProcessUtilization FROM ( SELECT [timestamp], CONVERT(xml, record) AS [record] FROM sys.dm_os_ring_buffers WHERE ring_buffer_type = N'RING_BUFFER_SCHEDULER_MONITOR' ) AS x ) AS y
ORDER BY record_id DESC;

🔹 Étape 6 : Optimiser la requête

Appliquez les solutions adaptées en fonction de la problématique identifiée.

  • Ajoutez ou optimisez les index.
  • Réécrivez la requête pour éviter les fonctions scalaires dans WHERE.
  • Utilisez des requêtes paramétrées pour éviter le Paramètre Sniffing.
  • Limitez le parallelisme si nécessaire.
  • Optimisez tempdb (plusieurs fichiers de données, disques rapides).

🔹 Étape 7 : Vérifier tempdb

Assurez-vous que tempdb est correctement configuré.

-- Vérifier l'utilisation de tempdb
SELECT
DB_NAME(database_id) AS DatabaseName,
file_id,
file_type = CASE type_desc
WHEN 'ROWS' THEN 'Data File'
WHEN 'LOG' THEN 'Log File'
ELSE type_desc
END,
size_mb = size / 128.0,
used_mb = FILEPROPERTY(name, 'SpaceUsed') / 128.0,
free_mb = (size - FILEPROPERTY(name, 'SpaceUsed')) / 128.0
FROM sys.dm_os_buffer_descriptors
WHERE database_id = 2; -- tempdb a l'ID 2

🔹 Étape 8 : Configurer Resource Governor (si nécessaire)

Utilisez Resource Governor pour limiter les ressources par groupe de travail.

-- Créer un pool de ressources pour les requêtes ad-hoc
CREATE RESOURCE POOL AdHocPool WITH ( MAX_CPU_PERCENT = 30, CAP_CPU_PERCENT = 50, MAX_MEMORY_PERCENT = 20 );

-- Créer un groupe de travail
CREATE WORKLOAD GROUP AdHocGroup USING AdHocPool;

Script 2 : Index manquants

SELECT
migs.avg_total_user_cost * (migs.avg_user_impact / 100.0) * (migs.user_seeks + migs.user_scans) AS improvement_measure,
'CREATE INDEX [missing_index_' + CONVERT (varchar, mig.index_group_handle) + '_' + CONVERT (varchar, mid.index_handle) + '_' + LEFT (PARSENAME(mid.statement, 1), 32) + ']'
+ ' ON ' + mid.equality_columns +
ISNULL (' INCLUDE (' + mid.inequality_columns + ')', '') +
ISNULL (' WHERE ' + mid.predicate, '') AS create_index_statement,
migs.*, mid.database_id, mid.[object_id]
FROM sys.dm_db_missing_index_groups mig
INNER JOIN sys.dm_db_missing_index_group_stats migs ON migs.group_handle = mig.index_group_handle
INNER JOIN sys.dm_db_missing_index_details mid ON mig.index_handle = mid.index_handle
WHERE migs.avg_total_user_cost * (migs.avg_user_impact / 100.0) * (migs.user_seeks + migs.user_scans) > 10
ORDER BY migs.avg_total_user_cost * migs.avg_user_impact * (migs.user_seeks + migs.user_scans) DESC;
