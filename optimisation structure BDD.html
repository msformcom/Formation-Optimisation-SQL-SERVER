📖 Introduction

L'optimisation des structures de tables, des index et des statistiques est essentielle pour garantir des performances optimales dans SQL Server. Une base de données bien conçue peut réduire les temps de réponse des requêtes de 90% ou plus, tout en minimisant l'utilisation des ressources (CPU, I/O, mémoire).

Ce guide vous explique comment :

  • 🔹 Concevoir des tables efficaces (choix des types de données, normalisation, partitionnement).
  • 🔹 Créer et maintenir des index optimaux (clustered, non-clustered, filtrés, columnstore).
  • 🔹 Gérer les statistiques pour aider le Query Optimizer à générer les meilleurs plans.
  • 🔹 Surveiller et diagnostiquer les problèmes de performance liés aux structures.

🗃️ Optimisation des Structures de Tables

La conception de vos tables a un impact direct sur les performances de vos requêtes. Une table mal conçue peut entraîner des lectures inutiles, une fragmentation excessive, ou des opérations de jointure coûteuses.

1. Choix des Types de Données

Le choix du type de données pour chaque colonne influence :

  • 🔹 L'espace de stockage (ex. : INT vs BIGINT).
  • 🔹 La vitesse des opérations (ex. : DATETIME2 vs DATETIME).
  • 🔹 La précision (ex. : DECIMAL(10,2) vs FLOAT).

🔹 Types de données recommandés

Catégorie Type recommandé Type à éviter Raison
Entiers INT (4 octets) BIGINT (8 octets) Utilisez BIGINT uniquement si nécessaire (valeurs > 2 milliards).
Entiers petits SMALLINT (2 octets) ou TINYINT (1 octet) INT Économise de l'espace pour les petites valeurs.
Décimaux DECIMAL(p,s) ou NUMERIC(p,s) FLOAT ou REAL FLOAT peut causer des erreurs d'arrondi.
Chaînes de caractères VARCHAR(n) ou NVARCHAR(n) TEXT ou NTEXT TEXT est obsolète et moins performant.
Dates et heures DATETIME2 (8 octets) DATETIME (8 octets) DATETIME2 a une meilleure précision (100 ns vs 3,33 ms).
Booléens BIT (1 octet) TINYINT ou CHAR(1) BIT est optimisé pour les valeurs booléennes.

🔹 Exemples de choix de types de données

-- ✅ Bonne pratique : Utiliser INT pour les identifiants
CREATE TABLE Clients ( Id INT PRIMARY KEY, -- 4 octets, suffisant pour 2 milliards de valeurs Nom VARCHAR(100), -- Chaîne de longueur variable Age TINYINT, -- 1 octet, suffisant pour 0-255 Solde DECIMAL(10,2), -- Précision pour les montants monétaires DateNaissance DATE, -- 3 octets, pour les dates seulement EstActif BIT -- 1 octet pour un booléen );
-- ❌ Mauvaise pratique : Types de données trop grands
CREATE TABLE Clients ( Id BIGINT PRIMARY KEY, -- 8 octets, inutile si les valeurs sont < 2 milliards Nom NVARCHAR(100), -- 2 octets par caractère, inutile si pas de Unicode Age INT, -- 4 octets, trop grand pour un âge Solde FLOAT, -- Imprécis pour les montants monétaires DateNaissance DATETIME -- 8 octets, moins précis que DATETIME2 );

Bonnes pratiques pour les types de données :

  • 🔹 Utilisez le type le plus petit possible pour économiser de l'espace.
  • 🔹 Évitez TEXT, NTEXT, et IMAGE (obsolètes, utilisez VARCHAR(MAX), NVARCHAR(MAX), ou VARBINARY(MAX)).
  • 🔹 Utilisez DATETIME2 au lieu de DATETIME pour une meilleure précision.
  • 🔹 Utilisez DECIMAL ou NUMERIC pour les montants monétaires (pas de FLOAT).
  • 🔹 Utilisez VARCHAR au lieu de CHAR pour les chaînes de longueur variable.

2. Normalisation des Tables

La normalisation est un processus de conception de base de données qui vise à :

  • 🔹 Éliminer les redondances (données dupliquées).
  • 🔹 Améliorer l'intégrité des données (contraintes de clé).
  • 🔹 Simplifier les requêtes (moins de jointures inutiles).

🔹 Niveaux de normalisation

Niveau Description Avantages
1FN Chaque colonne contient des valeurs atomiques (pas de listes). Élimine les redondances de base.
2FN Toutes les colonnes non-clé dépendent de la clé entière (pas de dépendances partielles). Élimine les redondances liées aux clés composites.
3FN Aucune colonne non-clé ne dépend d'une autre colonne non-clé. Élimine les dépendances transitives.

🔹 Exemple de normalisation

-- ❌ Avant la normalisation (redondance)
CREATE TABLE Commandes ( Id INT PRIMARY KEY, ClientId INT, ClientNom VARCHAR(100), -- Redondance : Nom du client ProduitIds VARCHAR(100), -- Liste de produits (non atomique) MontantTotal DECIMAL(10,2) );
-- ✅ Après la normalisation (3FN)
-- Table Clients
CREATE TABLE Clients ( Id INT PRIMARY KEY, Nom VARCHAR(100) );

-- Table Produits
CREATE TABLE Produits ( Id INT PRIMARY KEY, Nom VARCHAR(100), Prix DECIMAL(10,2) );

-- Table Commandes
CREATE TABLE Commandes ( Id INT PRIMARY KEY, ClientId INT FOREIGN KEY REFERENCES Clients(Id), MontantTotal DECIMAL(10,2) );

-- Table LignesCommandes
CREATE TABLE LignesCommandes ( Id INT PRIMARY KEY, CommandeId INT FOREIGN KEY REFERENCES Commandes(Id), ProduitId INT FOREIGN KEY REFERENCES Produits(Id), Quantité INT, PrixUnitaire DECIMAL(10,2) );

Bonnes pratiques pour la normalisation :

  • 🔹 Normalisez jusqu'à la 3FN dans la plupart des cas.
  • 🔹 Dénormalisez si les jointures deviennent trop coûteuses (ex. : tables de reporting).
  • 🔹 Utilisez des vues indexées pour combiner les avantages de la normalisation et de la dénormalisation.

3. Partitionnement des Tables

Le partitionnement permet de diviser une table en plusieurs parties physiques (appelées partitions), tout en la traitant comme une seule table logique.

🔹 Avantages du partitionnement

  • ⚡ Amélioration des performances : SQL Server peut ignorer les partitions non pertinentes.
  • 💾 Gestion simplifiée : Sauvegarde, restauration, et maintenance par partition.
  • 🔄 Chargement et archivage plus rapides : Ajout/suppression de partitions sans impact sur les autres.

🔹 Implémentation du partitionnement

-- Étape 1 : Créer une fonction de partitionnement (par année)
CREATE PARTITION FUNCTION PF_ParAnnee (DATE) AS RANGE RIGHT FOR VALUES ( '2020-01-01', '2021-01-01', '2022-01-01', '2023-01-01' );

-- Étape 2 : Créer un schéma de partitionnement
CREATE PARTITION SCHEME PS_ParAnnee AS PARTITION PF_ParAnnee TO (FG_2020, FG_2021, FG_2022, FG_2023, FG_DEFAULT);

-- Étape 3 : Créer la table partitionnée
CREATE TABLE Commandes ( Id INT IDENTITY(1,1), ClientId INT, DateCommande DATE, Montant DECIMAL(10,2) ) ON PS_ParAnnee(DateCommande);

Bonnes pratiques pour le partitionnement :

  • 🔹 Partitionnez sur une colonne fréquemment filtrée (ex. : date, ID de région).
  • 🔹 Utilisez des groupes de fichiers séparés pour chaque partition.
  • 🔹 Évitez de partitionner sur des colonnes avec peu de valeurs distinctes.

🔍 Optimisation des Index

Les index améliorent la vitesse des requêtes en permettant à SQL Server de trouver rapidement les lignes correspondantes à une condition (ex. : WHERE Id = 123).

1. Types d'Index

🔹 Clustered Index

Un index clustered détermine l'ordre physique des données dans la table. Il est obligatoire pour chaque table (par défaut, c'est la clé primaire).

CREATE CLUSTERED INDEX IX_Clients_Id ON Clients(Id);

🔹 Non-Clustered Index

Un index non-clustered est une structure séparée qui pointe vers les données de la table.

CREATE NONCLUSTERED INDEX IX_Clients_Nom ON Clients(Nom);

🔹 Index Composite

Un index composite est un index sur plusieurs colonnes.

CREATE NONCLUSTERED INDEX IX_Clients_Ville_Nom ON Clients(Ville, Nom);

🔹 Index Couvrant

Un index couvrant contient toutes les colonnes nécessaires pour une requête.

CREATE NONCLUSTERED INDEX IX_Clients_Ville ON Clients(Ville) INCLUDE (Nom, Prénom);

🔹 Index Filtré

Un index filtré ne contient que les lignes répondant à une condition.

CREATE NONCLUSTERED INDEX IX_Clients_Actifs ON Clients(Id) WHERE IsActive = 1;

Bonnes pratiques pour les index :

  • 🔹 Indexez les colonnes utilisées dans WHERE, JOIN, ORDER BY, GROUP BY.
  • 🔹 Utilisez des index composites pour les requêtes filtrant sur plusieurs colonnes.
  • 🔹 Utilisez INCLUDE pour créer des index couvrants.
  • 🔹 Évitez les index redondants.

2. Maintenance des Index

🔹 Vérifier la fragmentation

SELECT OBJECT_NAME(ind.OBJECT_ID) AS TableName, ind.name AS IndexName, indexstats.avg_fragmentation_in_percent FROM sys.dm_db_index_physical_stats(DB_ID(), NULL, NULL, NULL, NULL) indexstats INNER JOIN sys.indexes ind ON ind.object_id = indexstats.object_id AND ind.index_id = indexstats.index_id WHERE indexstats.avg_fragmentation_in_percent > 10 ORDER BY indexstats.avg_fragmentation_in_percent DESC;

🔹 Réorganiser ou reconstruire un index

ALTER INDEX IX_Clients_Nom ON Clients REORGANIZE;

ALTER INDEX IX_Clients_Nom ON Clients REBUILD;

Bonnes pratiques pour la maintenance des index :

  • 🔹 Réorganisez les index avec une fragmentation entre 10% et 30%.
  • 🔹 Reconstruisez les index avec une fragmentation > 30%.

📊 Optimisation des Statistiques

Les statistiques aident le Query Optimizer à générer des plans d'exécution optimaux.

1. Création et Mise à Jour des Statistiques

🔹 Mise à jour automatique

ALTER DATABASE [MaBase] SET AUTO_UPDATE_STATISTICS ON;

🔹 Mise à jour manuelle

UPDATE STATISTICS Clients;

EXEC sp_updatestats;

Bonnes pratiques pour les statistiques :

  • 🔹 Activez AUTO_UPDATE_STATISTICS.
  • 🔹 Mettez à jour les statistiques manuellement après un chargement massif.

2. Impact des Statistiques sur le Query Optimizer

Les statistiques obsolètes peuvent entraîner des plans d'exécution sous-optimaux.

🔹 Détecter les statistiques obsolètes

SELECT OBJECT_NAME(s.object_id) AS TableName, s.name AS StatisticsName, s.stats_date(s.object_id, s.stats_id) AS LastUpdated FROM sys.stats s WHERE DATEDIFF(DAY, s.stats_date(s.object_id, s.stats_id), GETDATE()) > 7;
